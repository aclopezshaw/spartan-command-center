# V0.8.4 Review Handoff

## Purpose

This handoff captures the current state of the V0.8.4 review board and the broader product-architecture decisions made during the review. It is intended to allow a repo-aware ChatGPT project to resume the review without reconstructing the full conversation history.

The core review objective evolved from a narrow V0.8.4 stabilization audit into a full product-architecture review covering:

- milestone and release structure
- epic and ticket hierarchy
- product philosophy
- progression-system design
- future feature sequencing
- documentation and implementation workflow

The review is not complete, but the major architectural direction is now substantially clearer.

---

# 1. Reviewed Areas

## 1.1 Engineering and Product Governance

Reviewed:

- milestone philosophy
- release philosophy
- epic philosophy
- ticket philosophy
- top-down planning hierarchy
- relationship between Notion, Jira, Confluence, GitHub, and the SDCB
- documentation standards
- decision records and engineering review artifacts
- need for a Product Architecture Map
- need for a Ticket Migration Matrix

Key outcome:

The project should be planned top-down rather than bottom-up.

Approved hierarchy:

```text
Milestone
  -> Release
    -> Exit Criteria / Intended Outcome
      -> Epic
        -> Ticket
```

The review board should not create tickets first and then attempt to infer release meaning afterward.

## 1.2 Current Product Identity

Reviewed:

- what Spartan is actually for
- whether future milestones were drifting toward “building a game”
- whether lore systems were becoming disconnected from the habit tracker
- whether current systems still served the original purpose

Key outcome:

Spartan is not primarily a game.

Spartan is a progression and motivation engine built around real-world habit execution.

The working north-star statement is:

> The Progression Engine is the product. Everything else exists to help the user complete real-world habits and make that progress feel meaningful.

A second operational test was adopted:

> Does this encourage the user to complete the habit tracker?

If a proposed feature does not either:

1. feed real-world performance data into the progression engine, or
2. reinforce the meaning and motivation of progression,

then it should be redesigned, deferred, or removed.

## 1.3 Current System Categories

Approved conceptual split:

### Production Systems

These directly collect or organize real-world activity.

Examples:

- Habit Tracker
- SMU
- Intel Reports
- Training Reports
- academic tracking
- assignment systems
- workout logging
- hydration logging
- reading logging

### Motivation Wrappers

These make real-world progress feel meaningful.

Examples:

- Achievements
- Service Record
- Campaign History
- Promotions
- Fireteam relationships
- Command Trust
- campaign lore
- events
- armor
- skill trees
- readiness progression

Both categories must feed the same central progression engine.

## 1.4 Milestone Roadmap

The roadmap was renumbered so Live Operations becomes milestone 1.0.

Approved high-level milestone sequence:

```text
0.6 - Infrastructure / Foundation
0.7 - Fireteam
0.8 - Command School
0.9 - Specialization School
1.0 - Live Operations
```

The prior milestone numbering should be treated as superseded.

## 1.5 Release Structure

Every release should have:

- one clear purpose
- one recognizable identity
- explicit exit criteria
- a coherent set of epics
- no unnecessary mixing of infrastructure and player-experience goals

A release that contains two distinct purposes should normally be split.

## 1.6 Epic Architecture

Epics should be derived from release outcomes.

Approved sequence:

```text
Define release outcome
  -> Define release exit criteria
    -> Identify major bodies of work
      -> Make each major body of work an epic
        -> Break each epic into tickets
```

A proposed cross-cutting epic category was discussed but not adopted.

## 1.7 Ticket Architecture

The TMM should be organized in this order:

```text
Release
  -> Epic
    -> Tickets
```

A ticket should not be added until its release and epic are known.

Search-before-create is now the default behavior for SDCB changes.

## 1.8 Achievement Engine

The current achievement philosophy supports two independent dimensions:

### Discipline

Consecutive completions.

### Persistence

Total lifetime completions.

The achievement engine should be generalized by cadence rather than creating separate daily and weekly engines.

Approved cadence direction:

- Daily
- Weekly
- future Monthly
- possible future Quarterly

# 2. Approved Design Decisions

## 2.1 Progression Engine Is the Product

Approved.

All future product decisions should be evaluated against the Progression Engine.

## 2.2 Everything Must Feed or Reinforce the Habit Tracker

Approved.

Every system must do one of the following:

- feed real-world data into habit tracking
- provide motivation to complete habits
- reflect long-term habit performance
- create meaningful consequences or rewards from habit performance

## 2.3 Milestones Represent Player Transformation

Approved.

Working interpretation:

- 0.6: the system becomes stable and trustworthy
- 0.7: the Individual becomes a Fireteam member
- 0.8: the Fireteam member becomes a leader
- 0.9: the leader becomes a specialist
- 1.0: the Spartan enters live operational service

## 2.4 Releases Must Have a Single Identity

Approved.

Each release should answer:

- What changes for the user?
- Why does this release exist?
- What must be true when it is complete?
- What new capability or meaning does it introduce?

## 2.5 Exit Criteria Before Epics

Approved.

## 2.6 Use “Individual” as the Pre-Fireteam Keyword

Approved.

The word “Candidate” was judged too broad because the player remains a Candidate throughout the entire Spartan Candidate Program.

For the initial progression stage, use the capitalized term:

> Individual

Approved examples:

- Individual progression loop is complete
- Individual achievements and milestones are complete
- Service Record reflects Individual completion

When the Fireteam milestone begins, the keyword upgrades to Fireteam or Fireteam Member.

## 2.7 Fireteam Members Are Motivational Companions, Not Multiplayer Participants

Approved.

The platform remains fundamentally individual.

## 2.8 Unit Cohesion as the Fireteam Progression Mechanic

Approved as the current working name.

Working mappings:

- Michael -> Physical Readiness
- Paige -> Intelligence
- Ellie -> Recovery
- Zoe -> Professionalism

Working relationship phases:

1. acquaintance
2. growing trust
3. dependable teammate
4. trusted with your life

Each relationship milestone can award readiness points.

## 2.9 Command Trust as the Command School Mechanic

Approved conceptually.

Habits increase Command Trust; sustained neglect decreases it gradually.

Constraints:

- small failures should not be catastrophic
- one missed habit should not destroy progress
- sustained neglect should create meaningful consequences
- trust must be recoverable

## 2.10 Event Density Increases by Milestone

### Fireteam

Working event structure:

- 2 minor events
- 1 major midpoint event
- 2 minor events
- 1 final major event
- 2 remaining buffer/make-up opportunities

Use Field Exercise and Debrief terminology.

### Specialization School

Working event structure:

- 3 minor academic events
- 1 major academic midpoint event
- 3 minor clinical events
- 1 final major clinical event

### Live Operations

Events become campaign checkpoints with branching story consequences.

## 2.11 Live Operations Are Progress-Based, Not Strictly Time-Limited

Approved.

Campaigns require target XP or progress, and continue until completed.

## 2.12 Armory Becomes the Home of the Skill Tree

Approved conceptually.

Graduation grants Spartan armor.

The Armory becomes the location for:

- armor progression
- skill tree
- skill points
- upgrades
- possible habit unlocks
- event modifiers
- campaign advantages

## 2.13 Search Before Creating SDCB Tickets

Approved operational rule.

# 10. Review Guardrails for the Next Chat

## Do Not

- invent unrelated game systems
- separate lore from habit performance
- create tickets before release placement is known
- create duplicate SDCB tickets
- create epics without exit criteria
- treat Fireteam members as multiplayer users
- restore old milestone numbering without approval
- populate Confluence before review decisions are final
- let a release contain multiple unrelated identities

## Always

- apply the Progression Engine lens
- ask how each feature motivates habit completion
- plan top-down
- search before creating tickets
- preserve the Individual -> Fireteam -> Leader -> Specialist -> Spartan transformation
- introduce a distinct motivational mechanic at each milestone
- use Field Exercise and Debrief terminology for Fireteam training
- separate production systems from motivation wrappers
- make failure recoverable but meaningful
- keep the product grounded in real-life execution
